# FABLE72126 — Incorporation Brief for ResolveVector

> **Purpose:** hand this to Fable (or any frontier planning model) as the source
> brief for *how to incorporate* the useful tools/ideas surfaced in the
> 2026-07-21 YouTube review (`072126YoutubeA.md`) into **ResolveVector (RV)**.
> This is a "what to do and where to look" document, not a final design. Each
> item includes the real GitHub repo, what to extract, how it maps to RV's
> existing architecture, and a suggested integration approach.

## RV context Fable needs (ground truth — do not re-derive)
- RV is a **local-first, single-user, lead-centric** coding workbench. One lead
  orchestrator owns the run; deterministic safety/verification/approval gates sit
  **outside** the model. Node.js backend + browser UI, `.vibe/` project storage,
  `runtime.sqlite` + `runs/<id>/events.jsonl`.
- Existing seams to build on: `src/backend/anchor.js` (edit anchoring),
  `lead_agent.js` (reasoning loop with `propose_chunks`/`stage_chunks`/`update_plan`/
  `ask_advisor`), the **advisor bridge** (Playwright, multi-vendor review), the
  `model_outcomes` ledger + `pickLeadModel` routing (Phase 5 first slice), and a
  scout warm-start + `benchmark-lead.js` (12-task) harness.
- Explicit boundaries: **no** multi-user/remote/parallel-fleet (Phase 6, deferred);
  embeddings should **not** become the default black-box retrieval.
- Licensing matters: RV is a product. Prefer **MIT/Apache** deps; treat AGPL as
  study-only unless the license is acceptable. Flag every license below.

## Tier 1 — incorporate / study first (highest fit)

### A. Hash-anchored edits + LSP/DAP — `can1357/oh-my-pi`
- **Repo:** https://github.com/can1357/oh-my-pi (MIT). Notable fork with harness
  telemetry/tests: https://github.com/az9713/oh-my-pi . Base framework "pi":
  https://github.com/earendil-works/pi
- **What to extract:** (1) **hash/content-anchored edits** — target an exact line
  via a content-hash anchor instead of resending old/new strings (avoids
  whitespace/syntax breakage, big token savings); (2) **LSP** integration for
  workspace-level structural refactors (rename/move across files, fix barrel &
  alias imports before writing to disk); (3) **DAP** (debugger adapter protocol)
  to attach a debugger to a broken process and read live state; (4) subagent
  orchestration patterns.
- **Maps to RV:** RV already has `anchor.js`. Task 1 for Fable: **diff RV's
  anchoring against oh-my-pi's hash-anchor approach** and decide whether to adopt
  content-hash anchoring in the patch pipeline (feeds review/apply gates). LSP and
  DAP are net-new lead tools (`refactor_symbol`, `debug_attach`) — high value,
  scope as separate features behind the existing gate model.
- **Approach:** study the edit encoder first (smallest, safest win); prototype an
  LSP tool against one language RV users hit most; DAP later.
- **Priority:** ★★★ (edits), ★★ (LSP), ★ (DAP).

### B. Provider gateway + quota-aware fallback — `diegosouzapw/OmniRoute`
- **Repo:** https://github.com/diegosouzapw/OmniRoute (MIT). npm: `omniroute`.
- **What to extract:** one OpenAI-compatible endpoint fronting many providers,
  **quota-aware 4-tier auto-fallback** (subscription → API key → cheaper → free),
  and a token-compression layer (RTK+Caveman). Also embeds local tool processes.
- **Maps to RV:** this is RV's provider abstraction + the Phase-5 "capability tags
  / routing config" open item, already built and battle-tested. Two options for
  Fable to weigh: (1) **adopt** OmniRoute as RV's provider layer (fastest), or (2)
  **borrow the fallback/routing design** into RV's own `pickLeadModel` so routing
  stays inside RV's inspectable event log. Recommendation: borrow the design, keep
  routing native so decisions remain replayable in `events.jsonl`.
- **Priority:** ★★★ (this is the near-term RV Tier-A routing feature).

### C. Codebase knowledge graph for scout context — `Graphify-Labs/graphify`
- **Repo:** https://github.com/Graphify-Labs/graphify . Related/compare:
  https://github.com/Phoenixrr2113/codebase-graph and
  https://github.com/tirth8205/code-review-graph
- **What to extract:** deterministic **tree-sitter AST → knowledge graph** of the
  whole repo (code + docs + SQL + configs), **no vector store**, every edge tagged
  EXTRACTED vs INFERRED, community detection to label subsystems, exposed as an
  MCP server (stdio per-dev or http shared).
- **Maps to RV:** direct upgrade path for RV's **scout/context selection** and a
  step beyond the `codebase_map.py`/`codebase_map_js.py` we built. Crucially it's
  **graph, not embeddings** — aligns with RV's `open-decisions.md` stance against
  default black-box embeddings. Fable should evaluate wiring Graphify (or its
  approach) as a scout recall tool that returns a structural map in one call.
- **License caution:** confirm Graphify's license before bundling; if
  unfavorable, replicate the tree-sitter-graph approach natively.
- **Priority:** ★★★.

### D. Cross-vendor review = model fusion — Friend-of-Me bridge + `lobehub/lobehub`
- **Repos / refs:** LobeHub reference architecture (78k★, provider runtime +
  plan-execute-review + mixed-model group chat): https://github.com/lobehub/lobehub .
  The "Friend-of-Me" Claude↔Codex MCP bridge from video `GNHqMyc95OM` had no
  confirmed public repo at review time — **Fable should locate/verify it** before
  relying on it; the *pattern* is what matters.
- **What to extract:** run two models from **different vendors** on the same
  proposal and have them audit each other ("they catch each other's lies"); a
  supervisor/executive group-chat where models debate then synthesize.
- **Maps to RV:** RV's **advisor bridge + critic escalation** already exist; this
  is the "advocate/judge over one proposal" open item. Implement as a fusion mode
  on the existing advisor seam: lead produces candidate → advisor(s) on a
  different provider review → lead reconciles → normal verify/approve gate.
- **Priority:** ★★★ (builds on shipped advisor bridge).

## Tier 2 — evaluate for specific subsystems

### E. Graph memory (memory read-side) — `topoteretes/cognee`
- **Repo:** https://github.com/topoteretes/cognee (Apache 2.0, ~12k★). Runs on
  local models; vector+graph pipeline.
- **Maps to RV:** candidate for RV's open item "does injected memory improve
  outcomes." Evaluate against RV's *own* memory system with the benchmark harness
  before adopting a dependency. Pair with the tiered-memory shape from Hermes
  (short `memory/user/soul` core + on-demand deep search).
- **Priority:** ★★ (evaluate, don't adopt blind).

### F. Local MTP serving (optional, currently not needed) — `youssofal/MTPLX`, `jundot/omlx`
- **Repos:** https://github.com/youssofal/MTPLX , https://github.com/jundot/omlx
- **Context:** you already run Qwen3-Coder-Next-8bit on vllm-mlx at ~76 tok/s.
  We concluded MTP isn't worth the second-server overhead *now*. Keep these as
  the reference if you later want native MTP (Qwen 3.5/3.6-A3B) as an RV `local`
  provider. **Do not action unless the speed need changes.**
- **Priority:** ☆ (parked).

### G. Analytics over RV's own data — `duckdb/duckdb`

- **Maps to RV:** query `runtime.sqlite` / `events.jsonl` / `model_outcomes` for
  benchmark trends and token-audit rollups without a warehouse. Read-only
  analytics companion; not part of the runtime path.
- **Priority:** ★ (nice-to-have observability).

### H. Retrieval quality (only if embeddings are added) — Nemotron 3 Embed
- **Ref:** NVIDIA Nemotron 3 Embed (8B / 1B / 1B-NVFP4) on Hugging Face —
  `nvidia/` org; #1 on RTEB, handles code, 32k chunks, open weights. Fable to
  confirm exact HF repo id.
- **Maps to RV:** only relevant if RV adds an *optional* embedding retrieval layer
  (against the default-black-box caution). Hold unless graph-only (Graphify)
  proves insufficient.
- **Priority:** ☆ (conditional).

## Suggested sequencing for Fable
1. **OmniRoute-style routing (B)** into `pickLeadModel` + capability tags — closes
   an in-progress Phase 5 item, unlocks cost-aware local/cloud routing.
2. **Hash-anchored edits (A)** — audit vs `anchor.js`, adopt if it reduces
   apply-time failures; measure with `benchmark-lead.js`.
3. **Graphify-style scout graph (C)** — replace/augment the static codebase map
   with a queryable tree-sitter graph tool for the lead.
4. **Fusion/advocate-judge (D)** on the advisor bridge — cross-vendor review mode.
5. Then evaluate memory (E) and analytics (G). Park F/H until a concrete need.

For each, the pattern is the same and matches RV's philosophy: **keep the
decision inside RV's inspectable event log and behind the existing
verify/approve gates** — borrow the design rather than bolt on an opaque service.
